📝 Developer Implementation & Bug Log (May 07)

Task: Stabilizing Portfolio Asset Allocation Pipeline

1. Issue: Asset Allocation returning null or OTHER

  • Bug/Error Faced: When triggering the POST /v1/portfolio-summary/{id}/recalculate endpoint, the assetAllocations field in the response was completely null. After initial fixes, it populated but categorized all stocks as OTHER (Unknown segment).
  • Root Cause:
    1. The asset allocation logic wasn't fully integrated into the synchronous processing pipeline.
    2. Historical trade objects in the database were missing their InstrumentInfo (or it was set to UNKNOWN). Because of this, the calculation engine didn't know if a symbol like "TCS" was a stock, option, or future.
  • How We Tackled It:
    • Injected the calculateAssetAllocation method directly into the processTradeDetailsWithObjects pipeline so it fires during every recalculation.
    • Enhanced the trade metrics refresher to actively look for missing InstrumentInfo and auto-populate it.
  • Files Changed: TradeProcessingServiceImpl.java
    • Logic Behind Change: We used the InstrumentInfo.fromRawSymbol(trade.getSymbol()) factory method to generate proper instrument details based purely on the ticker symbol.
    • Crucial Fix: We had to move this population logic outside of the if (metrics == null) block. Initially, trades that already had basic P/L metrics were skipping the instrument check. Moving it outside ensures every trade is evaluated and assigned a proper market segment (like EQUITY).
    • Code Impact: The system now accurately parses symbols and groups them. The API response now correctly outputs 100.00% STOCK for equity trades instead of dumping them into the OTHER category.

2. Issue: Compilation Type Mismatch in Tests

  • Bug/Error Faced: [ERROR] incompatible types: java.lang.String cannot be converted to am.trade.common.models.enums.AssetClass during mvn install.
  • Root Cause: The core data model was recently refactored to use a strict AssetClass Enum, but the persistence layer tests (PortfolioMapperTest.java) were still using legacy hardcoded strings like "Equity".
  • How We Tackled It:
    • We audited the test file and replaced all instances of string literals with proper enum references (e.g., AssetClass.STOCK).
  • Files Changed: PortfolioMapperTest.java
    • Logic Behind Change: Aligning the test suite's mock data with the updated, type-safe domain models to ensure a clean, green build process.

3. Issue: Stale Code Executing (Phantom Bugs)

  • Bug/Error Faced: We added log.info statements to debug the pipeline, but the logs never appeared in the console, and our code changes had no effect on the API.
  • Root Cause: The monorepo structure. spring-boot-devtools was running from the am-trade-app module and failed to automatically detect and recompile changes made in the am-trade-services module. It was continuously running stale .jar files from the local .m2 repository.
  • How We Tackled It:
    • Stopped relying on auto-reloads for cross-module changes. We ran mvn install -DskipTests from the root directory to force a full project sync before restarting the application.
